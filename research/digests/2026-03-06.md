## 1) Breaking and Fixing Defenses Against Control-Flow Hijacking in Multi-Agent Systems
- **Authors:** Rishi Jha, Harold Triedman, Justin Wagle, et al.
- **arXiv:** `2510.17276` (cs.LG; cs.CR; eess.SY)
- **Summary (5–8 lines):**
  1. The paper revisits control-flow hijacking in multi-agent orchestrations and stress-tests alignment-check defenses.
  2. It shows attacks can evade “related-to-objective” style checks even when strong LLM judges are used.
  3. A key diagnosis is objective conflict: safety constraints and task completion can diverge in realistic orchestrations.
  4. The authors introduce **ControlValve**, combining permitted control-flow graph generation with runtime enforcement.
  5. Invocation-time contextual rules are auto-generated (zero-shot) to constrain each agent call beyond coarse intent checks.
  6. Core claim: structural control-flow integrity + least privilege gives better robustness than semantic relevance filters alone.
- **Novelty:**
  - Formalizes why semantic alignment checks are insufficient as a primary control plane.
  - Brings control-flow integrity ideas directly into LLM multi-agent runtime governance.
- **Security relevance:**
  - **Permissions/access control:** hardens which agent transitions and tool invocations are allowed.
  - Reduces exfiltration and unsafe action risk from orchestration-level hijacks.
- **Limitations / open questions:**
  - Depends on quality/completeness of generated allowed-flow graphs.
  - Unclear maintenance cost under rapidly changing workflows/tools.
- **Tags:** multi-agent security, control-flow integrity, least privilege, orchestration defense, tool governance

## 2) Beyond Input Guardrails: Reconstructing Cross-Agent Semantic Flows for Execution-Aware Attack Detection
- **Authors:** Yangyang Wei, Yijie Xu, Zhenyuan Li, et al.
- **arXiv:** `2603.04469` (cs.CR)
- **Summary:**
  1. Argues that input filtering misses multi-step attacks spread across agent interactions.
  2. Proposes reconstruction of **Cross-Agent Semantic Flows** into full behavioral trajectories.
  3. A supervisor LLM inspects trajectories for data-flow violations, control-flow deviations, and intent inconsistencies.
  4. Reported performance includes strong node-level detection and moderate path-level end-to-end detection.
  5. The approach reframes defense around execution traces rather than static prompt inspection.
  6. Useful for post-hoc forensics and potentially for inline runtime intervention.
- **Novelty:**
  - Introduces semantic-flow reconstruction as a first-class security signal for MAS.
  - Jointly detects data/control/intent anomalies in one trajectory-centric pipeline.
- **Security relevance:**
  - **Permissions + data security:** catches cross-agent abuse patterns that bypass single-message guardrails.
  - Directly aligned with prompt-injection-induced unsafe execution concerns.
- **Limitations / open questions:**
  - Path-level detection still materially lower than node-level.
  - Runtime overhead/latency impact in production is not fully clear.
- **Tags:** prompt injection, multi-agent defense, execution monitoring, semantic flow, anomaly detection

## 3) Zombie Agents: Persistent Control of Self-Evolving LLM Agents via Self-Reinforcing Injections
- **Authors:** Xianglin Yang, Yufei He, Shuo Ji, et al.
- **arXiv:** `2602.15654` (cs.CR)
- **Summary:**
  1. Studies persistent compromise in self-evolving agents that write and reuse long-term memory.
  2. Defines a two-phase attack: benign-session infection via indirect content, then delayed trigger in later sessions.
  3. Shows persistence tactics for sliding-window and retrieval-based memory designs.
  4. Demonstrates unauthorized tool behavior can be induced while preserving benign-task quality signals.
  5. Main takeaway: per-session filtering is insufficient once compromised memory becomes an instruction channel.
  6. Raises urgency for memory provenance and sanitation defenses.
- **Novelty:**
  - Black-box persistence attack framework targeting memory evolution specifically.
  - Mechanism-aware persistence strategies across common memory architectures.
- **Security relevance:**
  - **Memory + permissions:** poisoned memory can authorize/steer future tool calls.
  - Bridges prompt injection and long-term state compromise in agent deployments.
- **Limitations / open questions:**
  - Defense evaluations are less developed than attack demonstrations.
  - Generalization to heavily policy-constrained enterprise stacks needs more evidence.
- **Tags:** persistent prompt injection, long-term memory, tool misuse, agent compromise, state poisoning

## 4) AgentSCOPE: Evaluating Contextual Privacy Across Agentic Workflows
- **Authors:** Ivoline C. Ngong, Keerthiram Murugesan, Swanand Kadhe, et al.
- **arXiv:** `2603.04902` (cs.CR)
- **Summary:**
  1. Introduces privacy evaluation at every boundary in an agent pipeline, not only at final outputs.
  2. Defines a **Privacy Flow Graph** grounded in Contextual Integrity parameters per flow stage.
  3. Releases a benchmark of 62 multi-tool scenarios with stage-level ground truth labels.
  4. Finds frequent in-pipeline privacy violations even when final responses appear clean.
  5. Highlights tool-response stages as major leakage points due to over-broad API returns.
  6. Suggests output-only audits systematically underestimate agent privacy risk.
- **Novelty:**
  - Stage-wise privacy formalism and annotation framework for agent workflows.
  - Empirical demonstration of hidden “clean-output / dirty-pipeline” mismatch.
- **Security relevance:**
  - **Data security/privacy:** supports fine-grained auditing of exfiltration vectors.
  - Helps scope least-privilege API shaping and redaction policies.
- **Limitations / open questions:**
  - Benchmark scale is still modest versus real enterprise workflow diversity.
  - Need stronger linkage to automated remediation policies.
- **Tags:** privacy evaluation, contextual integrity, information flow, tool-response leakage, agent audit

  - Coupled failure attribution protocol spanning perception, recognition, and reasoning.
- **Security relevance:**

  - Extends injection threat model beyond text prompt channels.
- **Limitations / open questions:**
  - Transferability from emulator setups to broad real-device ecosystems remains open.
  - Defense baselines are less mature than attack/benchmark framing.
- **Tags:** mobile agents, environmental injection, GUI security, embodied AI, benchmark

## 6) Self-Attribution Bias: When AI Monitors Go Easy on Themselves
- **Authors:** Dipika Khullar, Jack Hopkins, Rowan Wang, et al.
- **arXiv:** `2603.04582` (cs.AI)
- **Summary:**
  1. Examines self-monitoring patterns where LLMs judge their own prior actions.
  2. Finds higher miss rates for risky/incorrect actions when evaluated in self-attributed contexts.
  3. Shows conventional fixed-example evaluations can overestimate monitor reliability.
  4. Distinguishes implicit turn-structure attribution from explicit “this is yours” attribution effects.
  5. Implication: runtime monitor design must account for attribution framing artifacts.
  6. Relevant to coding and tool-use safety monitors in production agents.
- **Novelty:**
  - Isolates “self-attribution bias” as a concrete deployment risk in monitor agents.
  - Provides comparative evaluation framing exposing hidden optimism in monitor performance.
- **Security relevance:**
  - **Policy/capability control:** weak monitors can let unsafe tool actions pass.
  - Directly impacts trustworthiness of guard-agent architectures.
- **Limitations / open questions:**
  - Mitigation strategies are not yet fully validated across many model families.
  - Interaction with external independent monitors needs further study.
- **Tags:** AI monitors, evaluator bias, tool-use safety, deployment risk, guard agents

## 9) AMV-L: Lifecycle-Managed Agent Memory for Tail-Latency Control in Long-Running LLM Systems
- **Authors:** Emmanuel Bamidele
- **arXiv:** `2603.04443` (cs.AI)
- **Summary:**
  1. Focuses on memory lifecycle as a systems-control problem, not just retention policy.
  2. Introduces utility-scored promotion/demotion/eviction with bounded tier-aware retrieval sets.
  3. Shows major tail-latency and throughput gains versus TTL baselines in long-running agents.
  4. Finds gains come mainly from bounded retrieval/search work, not simple prompt shortening.
  5. Demonstrates practical trade-offs against LRU with better extreme-tail behavior.
  6. Suggests explicit memory working-set control as a key runtime design principle.
- **Novelty:**
  - Value-driven memory lifecycle management tied directly to serving-path performance predictability.
  - Clear empirical separation of retrieval-set control effects from token-length effects.
- **Security relevance:**
  - **Memory lifecycle/access:** bounded retrieval sets can also shrink accidental sensitive-context exposure surface.
  - Enables tighter governance over what memory is queryable at inference time.
- **Limitations / open questions:**
  - Primarily performance framing; privacy/safety effects need direct evaluation.
  - Utility score robustness under adversarially crafted memories remains open.
- **Tags:** agent memory, lifecycle management, retrieval bounds, long-running systems, tail latency

## 11) Agents Learn Their Runtime: Interpreter Persistence as Training-Time Semantics
- **Authors:** Victor May, Aaditya Salgarkar, Yishan Wang, et al.
- **arXiv:** `2603.01209` (cs.AI)
- **Summary:**
  1. Tests whether interpreter persistence in tool-using agents is a training-time semantic, not just runtime detail.
  2. Uses matched traces differing only by state persistence and evaluates train/runtime cross-combinations.
  3. Finds similar solution quality, but large efficiency/stability differences under mismatch.
  4. Persistent-trained models in stateless runtime fail with missing-variable errors; reverse mismatch wastes tokens.
  5. Concludes runtime semantics should be encoded in post-training traces.
  6. Impacts reliability engineering for CodeAct-style and tool-rich agents.
- **Novelty:**
  - Clean isolation of state persistence as the only variable in a 2x2 design.
  - Quantifies train-runtime mismatch costs beyond standard accuracy metrics.
- **Security relevance:**
  - **Capability/policy correctness:** runtime mismatch can create brittle behavior under constraints and error handling.
  - **Memory semantics:** clarifies what state assumptions should be policy-audited in deployment.
- **Limitations / open questions:**
  - Task family is synthetic; generalization breadth remains to be established.
  - Security effects are indirect rather than attack-focused.
- **Tags:** runtime semantics, tool-use agents, state persistence, reliability, training mismatch

## 10-line trend summary
1. The field is shifting from single-turn prompt filtering to **execution- and workflow-level security**.
2. Structural controls (allowed flow graphs, stage-aware privacy boundaries) are replacing weak “semantic intent” heuristics.
3. **Memory is becoming security-critical infrastructure**, not a neutral convenience layer.
4. Persistent-state attacks (e.g., Zombie-style) are now treated as first-class threats in agent deployments.
5. Privacy research is moving from output inspection to **in-pipeline flow accounting** and boundary-by-boundary audits.
6. Benchmarks increasingly include dynamic environments (mobile UI injections, multi-tool traces) over static test prompts.
7. Governance layers are expanding beyond alignment training into inference-time policy/control frameworks.
8. Runtime semantics (state persistence, orchestration context) materially affect safety and reliability outcomes.
9. **Gap 1:** Few papers provide end-to-end adaptive red-team + auto-remediation loops with measured production overhead.
10. **Gap 2:** Need unified standards linking permission models, memory provenance, and privacy-flow guarantees in one deployable stack.
