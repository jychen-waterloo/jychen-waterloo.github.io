### 1) Breaking and Fixing Defenses Against Control-Flow Hijacking in Multi-Agent Systems
- **Authors:** Rishi Jha, Harold Triedman, Justin Wagle, Vitaly Shmatikov  
- **arXiv:** 2510.17276v2 (**cs.LG**, cs.CR, eess.SY)
- **Summary (5-8 lines):**
  - Demonstrates that current alignment-check defenses (e.g., “is this step aligned with user goal?”) can be bypassed by control-flow hijacking attacks.
  - Shows attacks can induce unsafe agent-to-agent invocations while still appearing locally aligned.
  - Argues this is structural: functional flexibility and safety constraints are often in direct tension.
  - Introduces **ControlValve**, inspired by control-flow integrity + least-privilege principles.
  - ControlValve builds permitted control-flow graphs and enforces runtime trajectory conformance.
  - Adds contextual invocation rules per agent call to reduce latent privilege escalation paths.
- **Novelty:**
  - Reframes multi-agent prompt-safety as **control-flow integrity**, not only semantic alignment.
  - Combines graph-constrained orchestration with contextual invocation policy checks.
- **Security relevance:**
  - **Permissions/policy:** strong; explicit least-privilege and permitted-call graph enforcement.
  - **Data security:** addresses exfiltration risks caused by orchestrator-level hijacking.
- **Limitations / open questions:**
  - Zero-shot contextual rule generation may be brittle on unseen workflows.
  - Tradeoff between strict flow constraints and task coverage in dynamic environments.
- **Tags:** multi-agent security, control-flow integrity, least privilege, orchestration defense, exfiltration

### 2) AudAgent: Automated Auditing of Privacy Policy Compliance in AI Agents
- **Authors:** Ye Zheng, Yimin Chen, Yidan Hu  
- **arXiv:** 2511.07441v5 (**cs.CR**, cs.AI)
- **Summary (5-8 lines):**
  - Proposes a runtime auditing stack that checks whether agent behavior actually matches stated privacy policy.
  - Formalizes policy text into machine-checkable models via cross-LLM voting.
  - Performs runtime sensitive-data annotation and links events to policy constraints.
  - Uses ontology + automata checking for continuous compliance verification.
  - Adds execution-trace UI to make violations visible and attributable.
  - Reports real violations and insufficient policy guardrails for high-sensitivity data handling.
- **Novelty:**
  - End-to-end bridge from natural-language privacy policy → formal spec → runtime enforcement.
  - Continuous auditing architecture that is framework-agnostic and operationally practical.
- **Security relevance:**
  - **Data/privacy:** direct; detects and blocks policy-violating data operations.
  - **Audit/governance:** provides evidence trails for compliance and accountability.
- **Limitations / open questions:**
  - Policy formalization quality remains dependent on LLM parsing reliability.
  - Legal/policy heterogeneity across jurisdictions may reduce portability.
- **Tags:** privacy compliance, runtime audit, policy formalization, sensitive data control, agent governance

### 3) Adaptive Memory Admission Control for LLM Agents
- **Authors:** Guilin Zhang, Wei Jiang, Xiejiashan Wang, Aisha Behr, Kai Zhao, Jeffrey Friedman, et al.  
- **arXiv:** 2603.04549v1 (**cs.AI**, cs.CL, cs.MA)
- **Summary (5-8 lines):**
  - Targets a core failure mode in long-lived agents: indiscriminate memory growth and contamination.
  - Proposes explicit **memory admission control** to decide what gets written into long-term memory.
  - Emphasizes auditability and policy clarity versus opaque fully-LLM memory heuristics.
  - Frames admission as selective retention over salience, validity, and downstream utility.
  - Aims to reduce accumulation of stale/hallucinated/low-signal content.
  - Positions admission policy as a first-class control point in agent reliability.
- **Novelty:**
  - Treats memory-write gating as a dedicated systems component rather than incidental prompt logic.
  - Pushes auditable memory policy in place of purely generative retention decisions.
- **Security relevance:**
  - **Memory security:** directly reduces persistence of poisoned or low-integrity traces.
  - **Permissions/data:** implicit defense against long-horizon prompt-injection persistence.
- **Limitations / open questions:**
  - Admission criteria may under-retain rare but later-critical context.
  - Need adversarial evaluation under targeted memory-poisoning scenarios.
- **Tags:** agent memory, admission control, memory poisoning resistance, lifecycle governance, auditability

### 4) Memory as Ontology: A Constitutional Memory Architecture for Persistent Digital Citizens
- **Authors:** Zhenghui Li  
- **arXiv:** 2603.04740v1 (**cs.AI**, cs.MA)
- **Summary (5-8 lines):**
  - Argues memory should be treated as the ontological core of persistent agents, not just a retrieval module.
  - Proposes Constitutional Memory Architecture (CMA) with layered governance over memory operations.
  - Separates model identity from continuity-of-self stored in governed memory structures.
  - Introduces lifecycle framing for long-duration digital entities and cross-model persistence.
  - Compares against mainstream memory-tool paradigms as solving a different, narrower problem.
  - Emphasizes governance-before-functionality for persistent digital citizens.
- **Novelty:**
  - Conceptual shift from memory-as-tool to memory-as-identity substrate.
  - Governance hierarchy embedded into memory architecture design.
- **Security relevance:**
  - **Memory dimension:** high relevance for long-term integrity and continuity constraints.
  - **Policy/permissions:** constitutional layers can formalize who/what may mutate core memory.
- **Limitations / open questions:**
  - Strong conceptual framing; empirical security validation remains limited.
  - Unclear operational overhead in production-scale agent platforms.
- **Tags:** constitutional memory, persistent agents, identity continuity, memory governance, lifecycle design
